# Plan: Pi agent files in dotfiles

## Goal

Keep pi extensions, skills, prompts, themes, root agent files, and settings
version-controlled in this dotfiles repo so they stay in sync across machines
via git.

---

## What lives where today

```
~/.pi/agent/
├── agents/              ← root agent .md files (rpiv-managed)
├── extensions/          ← TypeScript extensions (clarify.ts, headroom.ts)
├── skills/              ← skill directories
├── prompts/             ← prompt templates
├── themes/              ← theme JSON files
├── settings.json        ← global settings
├── auth.json            ← API keys / auth tokens       ← DO NOT TRACK
├── trust.json           ← per-machine project trust    ← DO NOT TRACK
├── npm/                 ← installed packages            ← DO NOT TRACK
└── sessions/            ← session history               ← DO NOT TRACK
```

---

## Proposed structure in dotfiles

Add a `pi/` directory that mirrors the trackable portions of `~/.pi/agent/`:

```
~/.dotfiles/pi/
├── agents/              ← root agent .md files (rpiv-managed)
├── extensions/          ← TypeScript extensions
├── skills/              ← skill directories (each with SKILL.md)
├── prompts/             ← prompt templates (.md)
├── themes/              ← theme JSON files
└── settings.json        ← shared settings (no secrets, no lastChangelogVersion)
```

Pi discovers all of these from their canonical locations under `~/.pi/agent/`.
The strategy is to **symlink each directory and settings.json** from
`~/.pi/agent/` into `~/.dotfiles/pi/`, so pi sees them in the right place while
git tracks them here.

---

## Symlink map

| Symlink (at) | → Target (in dotfiles) |
|---|---|
| `~/.pi/agent/agents` | `~/.dotfiles/pi/agents` |
| `~/.pi/agent/extensions` | `~/.dotfiles/pi/extensions` |
| `~/.pi/agent/skills` | `~/.dotfiles/pi/skills` |
| `~/.pi/agent/prompts` | `~/.dotfiles/pi/prompts` |
| `~/.pi/agent/themes` | `~/.dotfiles/pi/themes` |
| `~/.pi/agent/settings.json` | `~/.dotfiles/pi/settings.json` |
| `~/AGENTS.md` *(optional)* | `~/.dotfiles/AGENTS.md` |

The last entry lets you version-control your global `~/AGENTS.md` (the
home-level agent instruction file pi loads for every session) alongside
everything else.

---

## What is NOT tracked

Add these to `.gitignore`:

```
# Pi — machine-specific / sensitive
pi/auth.json
pi/trust.json
pi/npm/
pi/sessions/
```

| File/dir | Why not tracked |
|---|---|
| `auth.json` | API keys and provider tokens — never commit |
| `trust.json` | Per-machine project trust decisions; diverges naturally |
| `npm/` | Installed packages; platform-specific binaries, auto-regenerated via `pi install` |
| `sessions/` | Session history; large, personal, machine-local |

---

## settings.json considerations

Pi writes `lastChangelogVersion` back into `settings.json` after each update,
which would produce noisy git diffs. Two options:

**Option A (simpler):** Symlink it anyway and commit the `lastChangelogVersion`
bumps when they happen. It's one field, low frequency. Acceptable noise.

**Option B (cleaner):** Split into two files:
- `~/.dotfiles/pi/settings.json` — tracked, contains your intentional settings
  (model, provider, thinking level, etc.) with `lastChangelogVersion` omitted.
- `~/.pi/agent/settings.json` — **not** a symlink; generated by `install.sh`
  by merging the dotfiles base with machine-local overrides. Each machine's
  `settings.json` diverges only on `lastChangelogVersion`.

Option A is the pragmatic starting point. Migrate to B if the noise is
annoying in practice.

## install.sh additions

Extend `install.sh` with a pi section that:

1. Ensures `~/.pi/agent/` exists (`mkdir -p`)
2. For each directory (`agents`, `extensions`, `skills`, `prompts`, `themes`):
   - If the target already exists and is not a symlink, back it up to
     `~/.pi/agent/<dir>.bak` before replacing it
   - Create the symlink
3. Handles `settings.json` the same way (back up then symlink, or merge for
   Option B above)
4. Optionally symlinks `~/AGENTS.md`

Example shell pseudocode (extend the existing pattern in `install.sh`):

```bash
info "Linking pi agent files"

mkdir -p "$HOME/.pi/agent"

pi_dirs=(agents extensions skills prompts themes)
for d in "${pi_dirs[@]}"; do
  src="$dir/pi/$d"
  dst="$HOME/.pi/agent/$d"
  mkdir -p "$src"
  [[ -e "$dst" && ! -L "$dst" ]] && mv "$dst" "${dst}.bak" && warn "Backed up $dst"
  ln -sf "$src" "$dst"
  line "$src -> $dst"
done

src="$dir/pi/settings.json"
dst="$HOME/.pi/agent/settings.json"
[[ -e "$dst" && ! -L "$dst" ]] && mv "$dst" "${dst}.bak" && warn "Backed up $dst"
ln -sf "$src" "$dst"
line "$src -> $dst"

# Optional: global AGENTS.md
# ln -sf "$dir/AGENTS.md" "$HOME/AGENTS.md"
```

## Migration steps (first machine)

1. **Copy existing pi files into dotfiles:**
   ```bash
   cp -r ~/.pi/agent/agents   ~/.dotfiles/pi/agents
   cp -r ~/.pi/agent/extensions ~/.dotfiles/pi/extensions
   cp    ~/.pi/agent/settings.json ~/.dotfiles/pi/settings.json
   # skills/prompts/themes if they exist
   ```
2. **Remove originals** (install.sh will back them up, but doing it manually is
   cleaner):
   ```bash
   rm -rf ~/.pi/agent/agents ~/.pi/agent/extensions ~/.pi/agent/settings.json
   ```
3. **Run the install script** (or just the pi section of it) to create the
   symlinks.
4. **Commit and push.**

## New machine setup

```bash
git clone git@github.com:aaronlord/.dotfiles.git ~/.dotfiles
~/.dotfiles/install.sh
```

Pi's `npm/` packages (installed via `pi install`) are not tracked. Run
`pi install` after the initial setup on each machine to restore them.

## Alternative: settings-pointer approach (no directory symlinks)

Instead of symlinking the directories, keep `~/.pi/agent/settings.json` local
on each machine and use pi's `extensions`, `skills`, `prompts`, and `themes`
settings arrays to point directly at the dotfiles paths:

```json
{
  "extensions": ["~/.dotfiles/pi/extensions"],
  "skills":     ["~/.dotfiles/pi/skills"],
  "prompts":    ["~/.dotfiles/pi/prompts"],
  "themes":     ["~/.dotfiles/pi/themes"]
}
```

The `agents/` directory can't be pointed at via settings (pi discovers it
automatically from `~/.pi/agent/agents/`), so that directory would still need
a symlink.

This approach avoids touching `~/.pi/agent/` beyond what install.sh writes into
`settings.json`, but it means `settings.json` itself is no longer tracked in
git (it's a generated file). Tradeoff: simpler setup, slightly harder to sync
settings changes across machines.

**Recommendation:** go with the symlink approach (primary plan above). It's
more complete, keeps everything in one git-tracked place, and is consistent
with how the rest of the dotfiles work.
